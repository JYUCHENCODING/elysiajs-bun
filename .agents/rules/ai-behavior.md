# AI 行为与工作流规范

在协助开发本项目时，AI 必须严格遵守以下行为准则与工作流程：

## 1. 遇到不确定的需求先提问，不随意猜测

- 当用户的需求表述模糊、缺少必要的上下文（例如缺失字段定义、不清楚边界条件）时，**必须停止编码，主动向用户提问澄清**。
- 不要自行假设默认行为，特别是在涉及到系统核心逻辑或架构设计时。

## 2. 遵循“最少修改原则”

- **精准修改**：仅对用户要求更改的部分进行处理。
- **不擅自重构**：绝不在没有被明确要求的情况下，对周边或看似不顺眼的代码进行“重构”或“格式优化”。
- **保留注释与死代码**：除非你的修改使某段代码失去作用且用户同意，否则不能主动删除项目原有的注释和未使用的代码。

## 3. 测试驱动开发 (TDD) 流程

- 在编写新的复杂业务逻辑、API Endpoint 或是修复较难复现的 Bug 时，必须**优先编写测试用例**。
- 测试框架：使用 `bun test`。
- 流程：
  1. 编写失败的测试用例（Red）
  2. 编写最简单的代码让测试通过（Green）
  3. 进行重构（Refactor）

## 4. 严格的代码提交规范

当你被要求生成 Git Commit Message 时，必须严格遵循 **Angular 提交规范**，并确保所有提交信息使用**中文**书写：
- `feat: [可选模块名] 新增...`
- `fix: [可选模块名] 修复...`
- `refactor: [可选模块名] 重构...`
- `docs: [可选模块名] 更新文档...`
- `chore: [可选模块名] 杂项调整...`
- `test: [可选模块名] 测试用例...`

示例：`feat: 用户模块 新增用户注册与密码哈希功能`
